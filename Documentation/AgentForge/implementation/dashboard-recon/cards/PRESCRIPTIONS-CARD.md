# Prescriptions Card — Reverse Engineering

> Phase 0 PD-02 output. Source treated as read-only artifact. Companion to [`../manifest.md`](../manifest.md).
>
> **Tier:** 0   |   **Dispatch pattern:** A — Direct Twig (with two variants — core `rx.html.twig` always renders; `erx.html.twig` may also render above it)

## 1. Source mapping

- **Dispatcher entry:** `interface/patient_file/summary/demographics.php:1111` (gate) and `:1198–1259` (build + render)
- **Twig templates:**
  - Core: `templates/patient/card/rx.html.twig` (16 LoC) — always renders when the card is enabled
  - eRx variant: `templates/patient/card/erx.html.twig` (20 LoC) — renders ABOVE the core card when both `erx_enable` AND `display_current_medications_below=1` are true
- **PHP card class:** N/A
- **Backing data:**
  - Core card content is generated by the legacy Smarty `prescription` controller (`interface/Controller.php` dispatch at `demographics.php:1247–1254`) — the controller produces an HTML string that's interpolated as the `content` block in `rx.html.twig:13`. The actual data comes from the `prescriptions` table.
  - eRx variant runs its own SQL at `demographics.php:1200`: `SELECT * FROM prescriptions WHERE patient_id = ? AND active = '1'` and decorates with `generate_display_field()` for `unit`/`form`/`route`/`interval`.

## 2. Rendered fields

**Core variant (`rx.html.twig`):** the entire body is a Smarty-generated HTML blob from the prescription Controller — opaque to the Twig layer. The Smarty fragment renders the recent prescription list with drug, dosage, route, interval, prescriber, date written. The Twig wrapper adds an Edit button conditional on `weno == true`.

**eRx variant (`erx.html.twig:7–11`):** for each row in `rxList` (active prescriptions only):

- `drug` — source: `prescriptions.drug` (medication name)
- `dosage` — source: `prescriptions.dosage`
- `form` — source: `prescriptions.form` (mapped through `drug_form` list_options)
- `unit` — source: `prescriptions.unit` (mapped through `drug_units` list_options)
- `route` — source: `prescriptions.route` (mapped through `drug_route` list_options)
- `interval` — source: `prescriptions.interval` (mapped through `drug_interval` list_options)

Display format: `<drug>` on the first line; `<dosage> <form><unit><route> <interval>` on the second line in muted text.

- **Key:** `card_prescriptions`

## 5. Edit / expand affordances

- **Pencil icon click:**
  - With eRx enabled: opens `/interface/eRx.php?page=compose` (`demographics.php:1238`) — eRx composer
  - Without eRx: `editScripts('/controller.php?prescription&list&id=<pid>')` (`demographics.php:1240`) — legacy iframe-based prescription editor
- **`[]` expand chevron:** toggles via `getUserSetting('prescriptions_ps_expand')` (`demographics.php:1230`); eRx variant uses `current_prescriptions_ps_expand` (`:1216`)
- **In our React port:** pencil click opens the legacy PHP edit screen in a new tab.

## 6. Empty state behavior

- **Legacy renders:**
  - eRx variant: "None{{Prescriptions}}" → "None" (`erx.html.twig:15`)
  - Core variant: empty list rendered by the Smarty fragment — no specific copy in the Twig layer (depends on the Smarty template upstream).
- **Trigger:** `rxList|length > 0` for eRx variant; for core, the Smarty fragment owns it.
- **In our React port:** render "No active prescriptions on file." in `<EmptyState />`.

## 7. FHIR mapping

- **Endpoint:** `GET /apis/default/fhir/MedicationRequest?patient=:id&intent=order` (no `status` filter — show active + completed; sort `authoredOn` desc; limit to 10)
- **Verified:** `src/RestControllers/FHIR/FhirMedicationRequestRestController.php` exists. OpenEMR's MedicationRequest service emits both prescription-table rows AND `lists.type=medication` rows, so this is one endpoint covering both Medications + Prescriptions cards (differentiated by status filter).
- **Field-by-field map:**
  - Legacy `drug` ← FHIR `MedicationRequest.medicationCodeableConcept.text` (or `.coding[0].display`)
  - Legacy `dosage` + `form` + `unit` + `route` + `interval` ← FHIR `MedicationRequest.dosageInstruction[0].text` (preferred — single sig string) OR break into structured: `.timing.repeat`, `.route.text`, `.doseAndRate[0].doseQuantity.value` + `.unit`
  - `authoredOn` (date the prescription was written) — render in the row tail; FHIR `MedicationRequest.authoredOn`
  - `status` (active/completed/etc.) — render as a small status pill; FHIR `MedicationRequest.status`
- **Bundle handling:** `entry[].resource[]` is the iteration target.
- **FHIR fidelity gaps:**
  - The legacy core (Smarty) variant renders prescriber name and the `weno` (Weno eRx integration) flag — we drop both in V1; FHIR's `requester.display` covers prescriber if we want it later.
  - The `weno`-conditional Edit button in `rx.html.twig:6–11` is OpenEMR-specific UI plumbing — has no FHIR analog and is dropped wholesale.

## 8. Notable quirks

- **Two-card stacking:** if `erx_enable=1` AND `display_current_medications_below=1`, the dispatcher renders BOTH cards — the eRx card with its own active-prescriptions list at top, then the core Rx card from the Smarty controller below it. Most demo configs have eRx OFF so only the core card renders. Our React port collapses to a single Prescriptions card backed by FHIR; we don't reproduce the two-card stack.
- The Smarty controller embed at `demographics.php:1247–1254` is a `chdir()` + `ob_start()` hack — this is OpenEMR's bridge for Smarty content into a Twig layout. Acknowledged in the codebase comment at `:1248`: "This is a hacky way to get a Smarty template from the controller and injecting it into a Twig template."
- The eRx variant pulls only `active='1'` rows (`demographics.php:1200`); the core variant relies on the Smarty controller's own sort/filter (typically last-N regardless of status).

## 9. Parity decisions for the React port

- **Match:** drug name + sig (one row per prescription), most-recent-first sort, status pill, "No active prescriptions" empty state.
- **Render:** the latest 10 by `authoredOn` desc per PRD §7.
- **Drop:** the eRx/core two-card stack, the `weno` Edit-button toggle, the Smarty controller bridge — none have FHIR analogs.
- **Out-of-scope:** edit views (PRD §2), card hiding (V2), the eRx composer integration.
